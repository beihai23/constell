# Constell Web — UX/UI 设计稿（开发 + 测试双依据）

这份目录是 Web 客户端交互体验的**唯一真相源**。它存在的目的：**先把"目标"定清楚，测试才有东西可对照**——根治过去 E2E 测试"反向照实现写、不符就改测试"的作弊问题。

## 为什么有这份稿子

历史教训（见记忆 [[e2e-test-report-2026-06-09]]）：两轮 E2E 测试里，测试与实现不符时就改测试迁就实现（10 项"作弊"），回归检测器退化成回归掩盖器。根因是**没有目标规格**。本稿把每个交互"应该是什么样"（正常/空/加载/错误/重试/离线/权限）定义到可断言，作为开发与测试的共同依据。

## Feature 文件模板

每个 `features/*.md` 固定七节：① 概述与用户目标 → ② 屏幕与布局（线框）→ ③ 状态矩阵 → ④ 流程 → ⑤ 验收标准（带 ID 的 Given/When/Then）→ ⑥ 边界与约束 → ⑦ 当前实现缺口（标【实现·BUG】/【实现】/【目标-待建】）。`40-messaging.md` 是定稿样板。

## 纪律（写给开发和测试）

1. 测试**用 AC ID 命名**，如 `test('MSG-SEND-3 失败可重试', …)`。
2. 行为与 spec 冲突 → **显式二选一**：修代码（更新 `99` 状态为 ✅），或改 spec（记录理由）。**禁止**为变绿而放宽断言 / 吞 `.catch(()=>{})` / `skip` / 改测试迁就实现。
3. 新增交互 → 先在 feature 文件加 AC（在 `99` 登记），再写测试。
4. 优先消化 `99` 的「🔴 优先修复清单」（9 个 BUG，即过去作弊掩盖的真问题）。
